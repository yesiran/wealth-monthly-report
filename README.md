# 叶思染财富月报 · 工作台

把「每月持仓分析报告」拆成 9 个步骤的流水线：确定性工作（行情、计算、图表、排版、PDF）全部由脚本完成，LLM（Opus 4.8）只做四类非它不可的事——识图、研究、写叙事、质检。步骤之间用文件交接，每步有校验门，改动脚本必须先过回归测试。

## 快速开始（每月一次）

```bash
cd 月报工作台
python3 run.py --period 2026-07
```

首次运行会自动创建本期目录、打开截图文件夹并在屏幕上告诉你下一步怎么做；把持仓截图放进去后**重新运行同一条命令**即可走完全流程（S0 识图后与 S7 出品后各停一次等你确认）。

```bash
# 只跑某几步 / 强制重跑 / 跳过确认：
python3 run.py --period 2026-07 --steps s4,s5 --force --yes
```

回归测试（改过 `scripts/` 后必须跑；`run.py` 每次启动也会自动先跑）：

```bash
python3 tests/run_tests.py           # 离线回归（黄金基线=2026-06）
python3 tests/run_tests.py --smoke   # 附加在线数据源冒烟
```

## 流水线

| 步骤 | 执行者 | 产物 | 校验 |
|---|---|---|---|
| S0 持仓识别 | LLM 识图 | `01_holdings.json` | 账户级盈亏=各股之和（±1元）；**人工确认点①** |
| S1 行情采集 | 脚本 | `02_market.json` + `raw/` 缓存 | 基金身份拟合（份额=金额÷净值，残差<0.03元）；期界覆盖 |
| S2 收益计算 | 脚本 | `03_returns.json` | 份额双算一致、分账户汇总闭合、权重=100% |
| S3 预判复盘 | LLM | `05_review.json` | 每条 verdict∈{✓✗◐⏳}+证据 |
| S4 研究 | LLM×9（联网） | `04_research/研究_*.md` | 章节完整、有日期来源、有「未查到」纪律 |
| S5 叙事 | LLM×(N+1) | `06_narrative/*.json` | 数字必须写 `{{token}}`、预判必须带验证窗口 |
| S6 出版 | 脚本 | `07_report.html` → `08_report.pdf` | 占位符全解析、关键数字在场、SVG 数量 |
| S7 质检 | LLM 读 PDF | `05_notes/qa_report.json` | blocker=0 才放行；**人工确认点②** |
| S8 档案回写 | LLM | `dossiers/*.md` 更新 | 每份档案新增本期小节 |

## 关键设计

- **识图的通用规则（S0）**：不为任何 App 写适配代码。提示词只规定一件事——提取「标的+份额」；份额不可见的基金抄下（金额/昨日收益/持仓收益）三个指纹数字，由代码反推份额。股票带 market 字段（hk/sh/sz/us），黄金、理财等暂不支持定价的资产如实记录为 gold/other 类型，S2 自动移入报告的「未纳入」部分——**识别永远不失败，只降级**。
- **研究任务自动规划（plan_research.py）**：不再有写死的标的清单。个股→一股一任务；指数基金→按 S1 抓到的"跟踪标的"自动分组（同指数共享研究）；主动/债基→宏观兜底任务。产物是每期的 `04_research/_plan.json`。
- **日志**：所有脚本双通道——控制台 INFO + 按日期落盘 DEBUG（`logs/YYYYMMDD/<步骤>_<时刻>.log`），每次运行独立文件；LLM 步骤的完整输出另存于各期 `05_notes/llm_*.log`。复盘按日期翻目录即可。

- **数字占位符**：全部报告数字只在 S2 产生一次；叙事里写 `{{h.03690.ret_m}}` 这类 token（词表见各期 `06_narrative/_tokens.txt`），出版时由脚本替换——模型永远不碰精确数字。
- **连续性（反截面式解读）**：`dossiers/` 每标的一份档案（论点/关键假设/预判记录/观点变化日志）。S4 研究"相对档案的增量"，S5 观点不许无证据漂移，S3 逐条复盘上期预判，S8 沉淀本期。观点改变必须写明"哪条假设破了"。
- **双数据源**：日K 东财→腾讯自动降级；汇率 东财中间价→frankfurter(ECB)降级（降级会打 flag，报告里注明口径）。基金净值按需翻页（单页行数会漂移，不做假设）。
- **基金身份拟合**：招行等 App 显示基金全称、与行情平台简称对不上；用截图三元组（金额/昨日收益/持仓收益）在候选里数值拟合确认真实代码，同时反推份额与买入日。code 未知就填 null，交给 S1。
- **黄金基线**：`2026-06/` 是首期（人工精调版），其数字经过手工双重验证，被冻结为回归测试的期望值（`tests/fixtures/`）。任何脚本改动若改变了这些数字，测试立刻报警。

## 目录

```
run.py                  驱动器（LLM 步骤经 claude -p headless 调用；首跑自动建目录并引导）
config/settings.json    模型、超时、确认点（无任何标的假设）
config/holdings_template.json  无上期数据时的默认结构（基准指数/汇率源）
config/report.css       报告样式
prompts/                每个 LLM 步骤一份作业规范
scripts/                lib(数据接口/日志) s1 s2 s6 plan_research validators topdf.js
dossiers/               标的档案（跨月持久，连续性核心）
logs/YYYYMMDD/          按日期归档的运行日志
tests/                  回归测试（70项）+ 黄金夹具
tests/baseline_2026-06/ 首期黄金基线全产物（工作区只放当期，历史基线归测试区）
```

## 常见维护

- **新增/清仓持仓**：S0 会自动反映；新基金 code 填 null 由 S1 拟合；`config/settings.json` 的 `research_jobs` 增删对应研究分工；S8 会自动建档/标记 closed。
- **数据源又变了**：先跑 `tests/run_tests.py --smoke` 定位哪个源挂了；lib.py 里加/换备用源，跑回归确认黄金数字不变。
- **报告改版式**：只动 `config/report.css` 与 `scripts/s6_build_html.py`，跑 T5 组装回归即可，不影响数字与内容层。
- **月度自动化**：跑顺之后可用 crontab 或 Claude Code 的 /schedule 在每月 1 日触发 `run.py --period $(date +%Y-%m 上月) --yes`（截图仍需人工投放，建议保留确认点）。

## 关于数据（公开仓库说明）

本仓库只包含**代码、提示词与流程**。持仓数据、黄金基线（`tests/baseline_2026-06/`、`tests/fixtures/`）、标的档案、运行日志均被 `.gitignore` 排除、保留在本地——因此 clone 本仓库后回归测试中依赖基线数据的用例无法直接运行，需用自己的首期数据重建基线（跑通一期后把产物冻结进 `tests/` 即可）。

## 已知边界

- 截图中无明细的小额资产（如零头美股、养老金专项账户）列为「未纳入」，在报告方法说明中披露。
- 报告月内若有基金分红送配（FHSP）或港股除净，S1/S2 会打 flag 要求人工确认收益口径（首期已验证 5/15 除净的处理方式：计入上月收益、本月记现金流）。
- 假设截图日到月末未调仓；截图越接近月末误差越小。
